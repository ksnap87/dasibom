## 🏠 전체 구조 한눈에 보기

```
사용자 폰 (앱)
    ↕ 인터넷
백엔드 서버 (Express)
    ↕
Supabase (데이터베이스 + 로그인)
```

**앱 → 서버 → 데이터베이스** 순서로 데이터가 흐릅니다.
예: 회원가입하면 → 서버가 받아서 → 데이터베이스에 저장

## 📱 화면 구성 (mobile/src/screens/)

### 1. AuthScreen (로그인/회원가입)
- 처음 앱을 켜면 나오는 화면
- 이메일 + 비밀번호로 회원가입 또는 로그인
- Supabase Auth가 자동으로 처리

### 2. QuestionnaireScreen (8단계 설문)
- 회원가입 직후 한 번만 하는 설문
- 8단계: 기본정보 → 관계목표 → 성격 → 취미 → 종교 → 건강/재정 → 가족 → 상대선호
- 완료하면 `questionnaire_completed = true` 로 저장됨

### 3. SuggestionsScreen (추천 상대 탭 🌸)
- 설문이 끝나면 나오는 메인 화면
- 서버가 호환성 점수(0~100점)를 계산해서 높은 순으로 보여줌
- "관심 있어요" → 상대도 좋아하면 → 매칭!
- "다음에" → 그 사람은 다시 안 나옴

### 4. MatchesScreen (매칭 탭 💌)
- 양쪽 다 "관심 있어요"를 누른 경우에만 여기 나타남
- 탭하면 채팅방으로 이동

### 5. ChatRoomScreen (채팅방)
- 매칭된 상대와 문자 주고받기
- Supabase Broadcast 채널로 실시간 메시지 수신 (카카오톡처럼 즉시 전달)
- 읽음 처리: 상대방이 읽으면 내 메시지 옆 "1" 뱃지가 사라짐

### 6. ProfileScreen (내 프로필 탭 👤)
- 내가 입력한 설문 내용 확인
- 로그아웃 버튼

## 🖥️ 서버 구조 (backend/src/)

### API란?
앱이 서버에게 "이 데이터 줘" 또는 "이 데이터 저장해"라고 요청하는 통로

### routes/ (요청 경로들)

| 파일 | 하는 일 |
|------|---------|
| `profiles.js` | 프로필 저장/조회, 관심 표현(좋아요) |
| `matches.js` | 추천 상대 목록, 매칭 목록 조회 |
| `messages.js` | 메시지 보내기/읽기 |

### utils/scoring.js (핵심 알고리즘)
두 사람의 설문 답변을 비교해서 0~100점 호환성 점수를 계산

| 항목 | 점수 비중 | 계산 방법 |
|------|----------|-----------|
| 관계 목표 | 25점 | 둘 다 결혼 원하면 만점, 다르면 감점 |
| 종교 | 20점 | 같은 종교면 높음, 중요도에 따라 조정 |
| 생활방식 | 20점 | 흡연/음주/운동 패턴이 비슷할수록 높음 |
| 가족 가치관 | 15점 | family_importance 숫자 차이로 계산 |
| 취미 | 10점 | 겹치는 취미가 많을수록 높음 |
| 사교 방식 | 10점 | 사교 활동 빈도가 비슷할수록 높음 |

## 🗄️ 데이터베이스 구조 (Supabase)

### profiles 테이블
유저 한 명의 모든 정보를 저장하는 곳

```
profiles
├── id (고유 번호, 로그인과 연결됨)
├── name (이름)
├── birth_year (출생연도)
├── gender (성별)
├── city (도시)
├── relationship_goal (관계 목표)
├── religion (종교)
├── hobbies (취미, 여러 개 저장 가능)
└── ... (설문의 모든 항목들)
```

### interests 테이블
"A가 B에게 관심 있다"는 기록

```
interests
├── from_user_id (관심 표현한 사람)
├── to_user_id (받은 사람)
└── is_liked (true=좋아요, false=다음에)
```

### matches 테이블
양쪽 다 좋아요 누른 경우 자동 생성

```
matches
├── user1_id
├── user2_id
└── compatibility_score (호환성 점수)
```

### messages 테이블
채팅 메시지 하나하나 저장

```
messages
├── match_id (어느 매칭의 채팅인지)
├── sender_id (보낸 사람)
├── content (메시지 내용)
└── read_at (읽은 시각)
```

## 🔐 보안 구조

- **anon key** → 앱에 넣어도 안전. 테이블별 RLS 규칙으로 제한됨
- **service_role key** → 서버(.env)에만 저장. 절대 앱 코드에 넣으면 안 됨
- **RLS (Row Level Security)** → "내 데이터는 나만 볼 수 있다" 규칙
  - 예: 내 채팅은 나와 상대방만 볼 수 있음

## 🚀 실행 방법

### 백엔드 실행
백엔드는 Railway에 항상 배포되어 있습니다.
- 주소: `https://dasibom-production.up.railway.app`
- 별도 실행 불필요

### 앱 실행 (Android 에뮬레이터)
```bash
# 1. Android Studio → Device Manager에서 에뮬레이터 실행
# 2. APK 빌드
cd ~/Claude/사이드잡/dasibom/mobile/android
export JAVA_HOME="/Applications/Android Studio.app/Contents/jbr/Contents/Home"
./gradlew assembleDebug

# 3. 설치
adb install -r app/build/outputs/apk/debug/app-debug.apk
```

### 에뮬레이터가 없으면?
1. Android Studio 설치
2. More Actions → Virtual Device Manager
3. Create Device → Medium Phone → API 36 선택 → 실행
